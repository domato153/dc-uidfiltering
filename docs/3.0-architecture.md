# DCUF 3.0 Architecture

> Historical refactor record — current instructions 아님. Use `AGENTS.md` and current source/build tooling for active work.

## Purpose
- Keep the existing mobile userscript behavior intact while we split the code into maintainable source files.
- Make future filter-rule changes portable to the PC-only script without copying large chunks by hand.

## Phase boundary
- `Phase 0`: freeze the current `v2.7.5.4` runtime contract and regression surface.
- `Phase 1`: keep behavior the same while building from split source files.
- `Phase 2+`: move storage/data/filter logic into shared modules.
- `Phase 4`: split mobile-only runtime into filter, personal-block, UI, and post-main fix modules while preserving original order in the build.

## Contract rules
- Shared filter code must not touch DOM directly.
- Target adapters are responsible for DOM traversal and visibility updates.
- Shared IP datasets live in one place only.
- Release builds keep `UTF-8 BOM`.
- Release builds force debug default to `OFF`.
- `@version`, file name, and init log string are synchronized at build time.

## Known high-risk seams
- Boot overlay and early hidden-body release timing.
- List filtering plus mirrored mobile cards sync.
- Comment filtering plus delayed reruns plus detached reply merge.
- Popup positioning and z-index fixes.
- Personal-block UI plus stored-data compatibility.

## Phase 5 observer notes
- Keep `FilterModule` container/body observers intact for now; Phase 5 only reduces duplicate rerun paths around them.
- `UIModule` no longer triggers comment typography normalization from its body observer. Comment typography stays owned by `post-main-fixes`.
- Full-page refilters are now coalesced into a single in-flight drain loop so settings, visibility recovery, and personal-block updates do not stack duplicate passes.
- Comment typography no longer uses a separate body `MutationObserver` only to bind `ResizeObserver` targets; target binding is folded into the normalization scheduler.
- Reply merge keeps the immediate and delayed merge passes, but only requests one stabilized comment refilter at the end of the burst.
- Residual manual risks: delayed comment rerenders on article pages, background-tab visibility recovery, mirrored list visibility sync, and popup-layer fixes after dynamic user-data popup changes.

## Phase 6 PC extraction notes
- The old PC script is no longer treated as the source of truth for future filter work.
- The future PC filter build should be derived from the mobile 3.0 shared filter stack plus a thin desktop adapter.
- `src/targets/pc/adapter-contract.js` now defines the adapter boundary, shared ownership, and the minimum hook surface needed for a desktop build.
- `docs/phase-6-pc-porting.md` is the extraction path document for turning mobile 3.0 filtering into a future PC filter script.
- `tools/build-pc-filter-userscript.mjs` is the current port rail: shared filter changes can flow into the desktop build through rebuilds before any desktop-specific DOM/UI edits are needed.
